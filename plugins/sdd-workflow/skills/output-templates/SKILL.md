# Output Templates Skill

Provides output format templates for AI-SDD workflow commands.

## Overview

This skill defines standard formats for Claude to output command execution results to users. Each template includes:

- Execution result summary
- Generated files/changes
- Next steps (actions for users to take)
- Recommended manual verification
- Verification commands

## Provided Templates

### 1. Initialization Complete Message (`init`)

**Template**: `templates/init_output.md`

**Usage**: Completion message after `/sdd_init` command

**Content**:

- Generated directory structure
- Created project principles file
- Generated template files
- Next steps (PRD creation, specification creation, etc.)

### 2. Task Breakdown Output (`breakdown`)

**Template**: `templates/breakdown_output.md`

**Usage**: Task list generated by `/task_breakdown` command

**Content**:

- Task list (by Phase)
- Dependency diagram (Mermaid)
- Implementation notes
- Reference documents

### 3. Cleanup Confirmation (`cleanup`)

**Template**: `templates/cleanup_output.md`

**Usage**: Confirmation message when executing `/task_cleanup` command

**Content**:

- Cleanup target confirmation
- Design decision integration status
- Files to be deleted

### 4. Specification Clarification Report (`clarification`)

**Template**: `templates/clarification_output.md`

**Usage**: Analysis results from `/clarify` command

**Content**:

- Clarity score
- List of unclear items (by category)
- Question list (prioritized)
- Implementation readiness criteria

### 5. Consistency Check Results (`check_spec`)

**Template**: `templates/check_spec_output.md`

**Usage**: Verification results from `/check_spec` command

**Content**:

- Consistency check results
- Details of inconsistencies
- Fix suggestions
- Implementation status updates

### 6. Migration Results (`migrate`)

**Template**: `templates/migrate_output.md`

**Usage**: Execution results from `/sdd_migrate` command

**Content**:

- Detected legacy configuration
- Executed migration process
- Next steps (verification items)

### 7. Constitution Management Results (`constitution`)

**Template**: `templates/constitution_output.md`

**Usage**: Execution results from `/constitution` command

**Content**:

- Principle creation/update results
- Compliance verification results
- Template sync status
- Next actions

### 8. PRD Generation Complete (`prd`)

**Template**: `templates/prd_output.md`

**Usage**: Execution results from `/generate_prd` command

**Content**:

- Generated PRD file
- CONSTITUTION.md compliance check results
- Next steps
- Recommended manual verification
- Verification commands

### 9. Specification & Design Doc Generation Complete (`spec`)

**Template**: `templates/spec_output.md`

**Usage**: Execution results from `/generate_spec` command

**Content**:

- Generated specification & design doc files
- CONSTITUTION.md compliance check results (spec + design)
- Next steps
- Recommended manual verification
- Verification commands

## Usage

From command md files, call as follows:

```markdown
## Output

Use the `output-templates` skill to output initialization complete message.
```

Or specify a particular template:

```markdown
## Output

Use the `breakdown` template from the `output-templates` skill to output task breakdown results.
```

## Template Structure

Each template has the following structure:

1. **Fixed Sections**: Output as-is (e.g., "### Generated Files")
2. **Placeholders**: Written in `{variable_name}` format, replaced with actual values at runtime
3. **Conditional Sections**: Output only under specific conditions (e.g., "Only when --full option is specified")

### Placeholder Types

| Placeholder | Description | Example |
|:--|:--|:--|
| `{feature-name}` | Target feature name | "user-login" |
| `{parent}` | Parent feature name (for hierarchical structure) | "auth/" |
| `{n}` | Count or number | "3" |
| `{task-name}` | Specific task name | "Implement user authentication API" |
| `{count}` | Count value | "5 items" |

## Updating Templates

When updating templates, note the following:

1. **Only user-facing information**: Not instructions for Claude, but messages for users to see
2. **Next steps should be specific**: Clearly state what users should do next
3. **Verification items in checklist format**: Use `- [ ]` notation
